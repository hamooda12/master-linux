# Lesson 07 — Viewing and Analyzing Service Logs with `journalctl`

---

# Overview

Knowing whether a service is running is only part of a Linux administrator's job.

In real production environments, services sometimes fail to start, crash unexpectedly, consume excessive resources, or produce unexpected behavior.

When this happens, administrators need to answer questions such as:

- Why did the service fail?
- When did it stop?
- What error message was generated?
- Did the configuration file contain an error?
- Has this happened before?

The answer is almost always found in **system logs**.

Modern Linux distributions using **systemd** store system and service logs in a centralized logging system called the **systemd journal**.

The primary tool used to access these logs is **`journalctl`**.

Learning how to read journal logs is one of the most valuable troubleshooting skills for Linux administrators and DevOps engineers.

# Why Logs Exist

Imagine this situation.

You restart the NGINX web server.

Instead of starting successfully, you receive:

```text
Job for nginx.service failed.
```

That message tells you **what happened**, but not **why**.

To discover the reason, Linux records detailed messages in its logs.

These logs may include:

- Configuration errors.
- Missing files.
- Permission problems.
- Network failures.
- Authentication attempts.
- Startup and shutdown events.

Without logs, troubleshooting would become little more than guessing.

---

# What is the System Journal?

The **system journal** is a centralized database that stores messages generated by:

- The Linux kernel.
- `systemd`.
- System services.
- User services.
- Applications.

Unlike traditional text log files, the journal is managed directly by `systemd` and can be searched efficiently using `journalctl`.

---

# What is `journalctl`?

`journalctl` is the command-line utility used to read entries from the system journal.

Instead of searching many different log files, administrators can query the journal using a single command.

Syntax:

```bash
journalctl [options]
```

---

# Viewing All Logs

To display every journal entry:

```bash
journalctl
```

Because the output is often very large, it opens inside a pager (`less`).

Useful navigation:

```text
Space      Next page
b          Previous page
/word      Search
n          Next match
q          Quit
```

---

# Viewing the Most Recent Logs

Display the latest journal entries:

```bash
journalctl -n 20
```

Example:

```bash
journalctl -n 50
```

This displays only the last 50 log messages.

---

# Viewing Logs for a Specific Service

This is one of the most common administrative tasks.

Syntax:

```bash
journalctl -u service_name
```

Example:

```bash
journalctl -u ssh
```

or

```bash
journalctl -u nginx
```

Now only messages related to that service are displayed.

---

# Following Logs in Real Time

To monitor logs as they are written:

```bash
journalctl -f
```

This behaves similarly to:

```bash
tail -f logfile
```

To follow one service:

```bash
journalctl -fu nginx
```

This is extremely useful while restarting or testing applications.

---

# Viewing Logs from the Current Boot

Sometimes a server has been running for months.

Older logs may make troubleshooting difficult.

To display only logs from the current boot:

```bash
journalctl -b
```

This removes entries from previous boots.

---

# Viewing Previous Boot Logs

To display the previous boot:

```bash
journalctl -b -1
```

Older boots:

```bash
journalctl -b -2
journalctl -b -3
```

This is useful when investigating why a server failed after a reboot.

---

# Filtering Logs by Time

Display logs since this morning:

```bash
journalctl --since today
```

Last hour:

```bash
journalctl --since "1 hour ago"
```

Specific time:

```bash
journalctl --since "2026-07-06 09:00"
```

Time range:

```bash
journalctl --since "09:00" --until "10:00"
```

---

# Viewing Only Errors

Show messages with high priority:

```bash
journalctl -p err
```

Critical messages:

```bash
journalctl -p crit
```

This helps administrators focus on important problems.

---

# Practical Scenario

Users report that SSH access suddenly stopped working.

First verify:

```bash
systemctl status ssh
```

Output:

```text
failed
```

Now investigate:

```bash
journalctl -u ssh
```

Recent entries reveal:

```text
Missing host key.
```

Instead of guessing, the administrator identifies the exact cause from the journal.

# Common Mistakes

## Mistake 1

Ignoring logs and immediately restarting services.

Always investigate the logs first.

---

## Mistake 2

Reading every log entry.

Use filters such as:

```bash
-u
-b
-n
-p
--since
```

to focus on relevant information.

---

## Mistake 3

Assuming the last error is the root cause.

Read the surrounding log messages to understand the sequence of events.

---

# Best Practices

- Check `systemctl status` first.
- Use `journalctl -u service` for service-specific troubleshooting.
- Follow logs in real time while testing changes.
- Filter by time to reduce unnecessary output.
- Investigate before restarting production services.
